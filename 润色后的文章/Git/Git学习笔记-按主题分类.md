## 1. Git基础概念

### Git仓库初始化
**问：** git init 命令的作用是什么？
**答：** 在现有目录中初始化一个新的Git仓库。

**问：** 执行git init命令会发生什么？
**答：** 创建.git目录，初始化Git本地仓库。

**问：** .git目录包含了哪些信息？
**答：** 项目的Git仓库信息，包括提交历史、分支、标签等。

**问：** .git目录的作用是什么？
**答：** 管理不同的版本，以及各个分支。记录所有的提交历史。

**问：** .git目录通常是可见的还是隐藏的？
**答：** 隐藏的。

**问：** 发现一个项目目录中有.git目录意味着什么？
**答：** 项目已被初始化为Git仓库，项目已经被添加到版本控制。

**问：** 如果一个项目没有.git目录会怎样？
**答：** 说明这个项目没有被版本控制。

**问：** 删除 .git 目录意味着什么？
**答：** 删除 .git 目录等于删除了整个 Git 仓库的历史记录。你的项目会变成一个普通文件夹，所有的分支、提交记录和版本信息都会丢失。

### Git用户配置
**问：** git config 命令的作用范围有哪两种？
**答：** 局部配置（单个仓库级别）和全局配置（用户级别/所有仓库）。

**问：** git config 命令配置局部用户名和邮箱的指令是什么？
**答：**
```bash
git config user.name "你的姓名或昵称"
git config user.email "你的邮箱地址"
```

**问：** git config 命令配置全局用户名和邮箱的指令是什么？
**答：**
```bash
git config --global user.name "你的姓名或昵称"
git config --global user.email "你的邮箱地址"
```

**问：** 使用--global参数和没有使用这个参数分别的作用是什么？
**答：** 没有使用 --global 参数的情况下运行 git config，Git 会将配置应用于当前仓库。使用 --global 参数时，Git 会将配置应用于当前用户的所有仓库。

**问：** 在Git中，配置用户名和邮箱的作用是什么？
**答：** 提交记录、代码审查、责任追踪里面会显示作者信息（用户名和邮箱）。

**问：** Git的用户名和邮箱配置，是否会影响你对远程仓库的访问权限？
**答：** 不会影响权限，仅影响你本地提交时的作者信息。

### Git安装和环境
**问：** Git必须安装在C盘吗？
**答：** 安装在哪个分区都行，只要正确配置了系统环境变量（PATH），Git 的命令行工具都可以正常工作。

**问：** 如何判断你的电脑是否已经安装了Git？
**答：** 在命令行输入 `git --version` 指令。

**问：** 下载Git后如何安装并测试是否成功？
**答：** 使用默认选项一路"Next"安装，完成后在终端输入 `git --version` 检查是否安装成功。

## 2. Git工作流程和区域管理

### Git三个工作区域
**问：** Git的工作流程中有哪三个区域？它们的作用是什么？
**答：**
- **工作区（Working Directory）：** 你实际编辑文件的地方
- **暂存区（Staging Area）：** 保存准备提交的修改
- **本地仓库（Local Repository）：** 存储所有提交历史的地方

**问：** 在命令行中执行git add。数据流向是什么？
**答：** 工作区更改添加到暂存区。

**问：** 执行了git commit。数据流向是什么？
**答：** 暂存区的更改提交到本地仓库。

**问：** 暂存区对应的英文是什么？
**答：** staging area

**问：** 当你执行git add命令时，更改会被添加到哪里？
**答：** 暂存区（staging area）。

### 工作状态检查
**问：** git status命令的作用是什么？
**答：** 查看当前工作区状态，包括文件修改、暂存情况，以及本地分支是否与远程分支同步。

**问：** git status会显示哪个区域的状态？
**答：** 显示当前工作目录和暂存区的状态。

**问：** git status输出中"Changes not staged for commit"表示什么？
**答：** 表示工作区中已修改但尚未添加到暂存区的文件。

**问：** git status输出中"Changes to be committed"表示什么？
**答：** 表示已放入暂存区、准备提交的文件，也就是执行过git add的内容。

**问：** git status输出中"Untracked files"表示什么？
**答：** 表示工作区中新建但未被Git跟踪的文件。

**问：** git status的绿色文字和红色文字分别表示什么？
**答：** 暂存的更改通常会以绿色文字显示，而未暂存的更改会以红色文字显示。

**问：** 在做任何操作（如commit、pull、push、merge）之前，先运行哪个指令是一个好习惯？
**答：** 运行`git status`，检查当前的变化处于哪个阶段（是仅仅修改了，还是已经准备好提交了）。

**问：** 运行git status返回以下信息表示什么？
```
On branch main
Your branch is up to date with 'origin/main'.
nothing to commit, working tree clean
```
**答：** 表示工作区、暂存区和最新提交完全一致，没有未提交的更改。

**问：** "working tree clean"意味着什么？
**答：** 表示暂存区、工作区和本地仓库内容完全一致。

**问：** 使用git指令添加文件的命令是什么？

**问：** 如何将本地仓库推到远程仓库？
**答：** `git push -u origin master`推送到GitHub。

**问：** 如何将本地项目推送到GitHub？请说明添加、提交、推送这三个git指令。

**问：** git diff命令通常用于什么？显示的是什么跟什么的差异？
**答：** 显示当前工作区与上次提交之间的差异，展示自上次提交以来对文件所做的未提交改动。

**问：** 默认情况下，git diff比较的是什么和什么之间的差异？
**答：** 工作目录中尚未暂存的修改，与暂存区中的内容。

**问：** 当自己对代码进行了修改，想要对比一下，相对于之前提交的版本，有哪些新的改变，要使用哪条git指令？
**答：** Show Diff

**问：** Show Diff在git中这个指令所显示的内容是什么？
**答：** 显示的是仓库中上次提交记录与当前工作区的区别。

### 基本操作
**问：** 执行哪个git命令时，更改会被添加到暂存区？
**答：** `git add`

**问：** git add命令的作用是什么？
**答：** 将更改的文件添加到暂存区（staging area）。

**问：** git commit命令的作用是什么？
**答：** 将暂存区的更改提交到本地仓库。

**问：** git push命令的作用是什么？
**答：** 将本地仓库的提交发送到远程仓库。

**问：** 如何将本地仓库与GitHub仓库链接？
**答：** 运行`git remote add origin [你的仓库的URL]`

**问：** 首次将本地仓库与GitHub仓库链接，git命令是什么？
**答：** `git remote add origin <https://github.com/your_username/your_repository.git>`

## 4. 分支管理

### 分支创建和切换
**问：** 新分支创建和切换的Git命令是什么？
**答：** `git switch -c <branch-name>`

**问：** git checkout -b feature/puyuyang_mock-interview中的-b是什么意思？
**答：** -b是--branch的缩写，表示在当前分支上创建新分支并立即切换到该分支。

**问：** git checkout -b xxx一步完成什么操作？
**答：** 创建 + 切换

**问：** 查看本地和远程所有分支的指令是什么？
**答：** `git branch -a`

**问：** 在新建并进入分支后，第一次推送时为什么要加上-u参数？
**答：** -u（或--set-upstream）用于将本地分支与远程分支关联，建立跟踪关系，之后可直接使用git push、git pull等命令而无需再指定远程分支。

### 分支合并
**问：** 如何合并分支？
**答：** 首先切换到目标分支（如main分支），然后执行`git merge <源分支名>`

**问：** 使用git merge命令合并分支时可能遇到的一个挑战是什么？
**答：** 可能会遇到代码冲突，需要手动解决。

**问：** 功能开发和Bug修复分支的命名风格是什么？
**答：**
- 功能开发：feature/<yourname>_<feature>，如 feature/puyuyang_login_api
- Bug修复：bugfix/<yourname>_<issue>，如 bugfix/puyuyang_fix_encoding

### 分支工作流
**问：** 开发时应该为每个功能单独建分支，还是把所有修改放在个人分支里？
**答：** 主流做法是每个功能一个分支，这种方式称为feature branch模型，是最常见且最安全的团队协作方式。

**问：** 为什么不建议把所有修改都放在个人分支里？
**答：** 因为无法部分合并，若他人只需功能A而功能B未完成，就不能单独合并。

**问：** Feature Branch Workflow这种开发模式是什么？
**答：** 每个人维护一个分支，一个分支对应一个功能，开发完以后就合并到主分支里面。每个开发人员为自己正在处理的功能创建一个新的分支，开发人员在自己的分支上进行代码编写提交到本地分支，并推送到远程仓库，然后将更改合并到主分支。

**问：** 为什么创建完分支后还要执行git push？
**答：** 因为创建的只是本地分支，远程仓库（如Codeup）并不知道该分支的存在。

**问：** git push -u origin feature/puyuyang_mock-interview做了哪两件事？
**答：**
① 将本地分支上传到远程仓库，生成同名分支 origin/feature/puyuyang_mock-interview
② 建立追踪关系，以后执行git push或git pull时可直接使用默认远程分支，无需再写origin和分支名

## 5. 远程仓库操作

### 克隆和远程连接
**问：** 如何初始化一个新的Git仓库？
**答：** 运行`git init`。

**问：** 如何把代码推送到GitHub？
**答：** 运行`git push -u origin master`或`git push -u origin main`。

**问：** 两个从远程仓库到本地仓库的git指令是什么？
**答：** pull和fetch

**问：** 如何将本地仓库的更改上传到远程仓库？
**答：** 通过推送（push）操作，使用git push命令。

**问：** 从远程仓库获取最新的更改有几种方式？
**答：**
- 一个指令：`git pull`
- 两个指令：`git fetch`和`git merge`

### 拉取操作
**问：** git pull命令的作用是什么？
**答：** 远程仓库的内容合并到本地仓库，从远程仓库获取最新的内容并尝试将其合并到本地仓库中。

**问：** git pull包含了哪些指令？
**答：** fetch和merge

**问：** git pull命令是哪两个命令的组合？
**答：** git fetch和git merge。

**问：** 在执行git pull之前，建议先做什么？
**答：** commit来提交到本地仓库。

**问：** 使用Git的拉取（pull）操作的目的？
**答：** 保持您的本地仓库与远程仓库同步，并减少合并冲突，获取到最新的代码状态。

**问：** 执行git pull的工作流程实际上包括哪两个步骤？
**答：** 获取远程仓库的更改（fetch），然后合并更改到当前分支（merge）。

**问：** 在git pull过程中如何处理存在冲突的更改？
**答：** 不会自动覆盖，而是让用户手动解决。

**问：** 如果远程仓库和本地仓库的更改没有冲突，执行git pull会发生什么？
**答：** Git会自动合并这些更改，本地代码不会被覆盖。

**问：** 当git pull遇到有冲突的更改时会怎样？
**答：** Git会停止合并过程，并让用户手动解决这些冲突。

### 推送操作
**问：** 我写完一个功能模块，想把它推到远程仓库，这个时候要执行哪三个git指令？
**答：** 首先执行commit提交到本地仓库，然后使用pull拉取远程仓库，最后push推送到远程仓库。

**问：** 如果远程仓库有新的更改，直接推送本地的更改到远程仓库会发生什么？
**答：** Git会要求先拉取（pull）远程更改，解决任何冲突，然后再次尝试推送push。

**问：** Git是否允许本地更改直接覆盖远程仓库的内容？
**答：** 可以覆盖远程仓库，但不建议。Git会试图保护远程仓库的历史不被意外覆盖，除非使用特殊操作如强制推送。

**问：** 在Git中，正确的操作顺序是什么？
**答：** 先commit到本地仓库，然后push到远程仓库。

## 6. 冲突解决和高级操作

### Rebase vs Merge
**问：** git pull --rebase和git pull有什么区别？分别等同于哪些指令组合？
**答：**
- `git pull = git fetch + git merge`
- `git pull --rebase = git fetch + git rebase`

**问：** git pull = git fetch + git merge的缺点是什么？
**答：** 它会生成合并提交（Merge commit），使提交历史混乱，充满类似"Merge branch 'main' of..."的信息。

**问：** git pull --rebase = git fetch + git rebase的优点是什么？
**答：** 它会先拉取远程代码，再将本地提交依次应用在其上，保持线性、干净的提交历史，避免多余的合并节点。

**问：** 为什么一直推荐使用git pull --rebase？
**答：** 因为它能保持提交历史线性、清晰，避免了大量不必要的"Merge branch..."提交记录，这在团队协作中能保持更加整洁的历史，便于追踪和理解每个提交的变动。

**问：** 在你提交了本地提交B，而远程已有提交C的情况下，执行git pull --rebase会发生什么？
**答：**
- fetch：Git先获取远程最新提交C，此时本地知道远程历史为A---C
- rebase：暂时收起你的本地提交B，使分支回到A
- 将本地分支快进到远程最新提交C
- 再把提交B应用到C之上，生成新的提交B'，内容与B相同，但父提交由A变为C

**问：** 在相同场景下，如果不使用--rebase，直接执行git pull会怎样？
**答：** Git会执行fetch + merge，结果会生成一个合并提交M，用于把本地提交B与远程提交C合并，不包含任何功能性修改。

### Stash操作
**问：** 使用git stash命令的目的是什么？
**答：** 临时保存工作区和暂存区的未提交的代码，临时保存工作区的未提交的修改，使工作区回到干净的状态。

**问：** Stash Changes作用是什么？
**答：** 允许你临时保存你的工作进度，而不会干扰到当前的工作流程。

**问：** Stash Changes作用是否是将更改存储到暂存区？
**答：** 不是，而是存储到Git的一个独立的stash区域。

**问：** git里面的暂存区和stash区域是一个东西吗？
**答：** 不是。

**问：** Stash Changes...数据流向是什么？
**答：** 将工作目录的更改保存到Git的Stash。

**问：** Stash Changes...使用场景是什么？
**答：** 当你正在进行的工作尚未完成且你需要切换到其他任务时，可以暂时存储更改。

**问：** git stash默认会同时保存哪里的修改？
**答：** 工作目录中的修改和暂存区中的修改。

**问：** git stash的设计目的是让开发者能够快速清理工作环境，方便切换分支或同步代码，同时保证什么？
**答：** 所有未提交的修改都被安全保存。

**问：** 本地修改未完成又不想提交时，应该用什么指令？作用是什么？
**答：** 使用`git stash`，它会暂存当前修改（含暂存区和工作区），并让工作目录恢复到上次提交（HEAD）状态。

**问：** 如果之前使用了Stash，想恢复修改，应使用什么指令？
**答：** 使用`git stash pop`。

### Reset操作
**问：** git reset命令的三种模式分别是什么？
**答：** --soft、--mixed和--hard

**问：** 使用git reset --hard HEAD~1命令的作用是什么？哪个仓库回到哪个状态？
**答：** 本地仓库回到上次提交之前的状态。

**问：** 已确定GitHub是最新正确版本，本地分支有问题，想以远程为主，应输入什么指令？
**答：** 使用`git fetch origin`再执行`git reset --hard origin/你的分支名`。

### Revert vs Force Push
**问：** 强制推送到远程仓库的Git命令是什么？
**答：** `git push --force`

**问：** 在团队合作项目中使用git push --force的副作用是什么？
**答：** 覆盖其他人的更改和重写提交历史，引起同步问题。

**问：** 使用git revert而不是git push --force的优势是什么？对于历史方面？
**答：** 避免重写公共历史，保留历史记录。

**问：** 团队合作中如何安全地处理错误的提交？使用的是什么git的指令？
**答：** `git revert`

## 7. IDE集成和SSH配置

### IDE快捷键和操作
**问：** IDE里面的快捷键，调出git操作？
**答：** alt + ~（左上角的符号）

**问：** Pycharm里面显示底部分支信息的快捷键？
**答：** alt + 9

**问：** WebStorm里面，当你点击"commit"按钮时，IDE实际上执行了哪2个指令？
**答：** 先git add然后git commit。

**问：** WebStorm里面，当你点击"commit"按钮时，IDE首先执行什么操作？这相当于执行什么命令。然后，IDE执行什么操作。这相当于执行了什么指令？
**答：** 首先将这些选中的更改添加到暂存区。这相当于在命令行中执行`git add`。然后，IDE将这些已经暂存的更改提交到本地仓库。这相当于执行了`git commit`。

**问：** WebStorm的Git集成中，提交（commit）对话框通常提供了一种便利，相当于合并了哪两个步骤？
**答：** 将更改添加到暂存区（staging area）和将暂存区的内容提交到仓库这两个步骤合并为一步执行。

**问：** 在WebStorm IDE中，"Rollback"选项的用途是什么？
**答：** 撤销最近的一次或多次Git提交。撤销的是本地仓库的提交。

**问：** Rollback在WebStorm中的作用类似于哪个Git命令？
**答：** `git reset --soft`

**问：** IDE里面，第一次如何将当前项目直接上传到GitHub或JetBrains Space？【使用到了哪个选项？】
**答：** 使用"Share Project on GitHub"或"Share Project on Space"选项。

**问：** webstorm里面，如果项目还没有启用Git版本控制应该选择什么？
**答：** "Enable Version Control Integration"然后选择"Git"，或者add to VCS。

### SSH配置和认证
**问：** SSH的英文全称以及中文名称是什么？
**答：** Secure Shell Protocol（安全外壳协议）

**问：** 生成SSH密钥对，密钥对指的是什么？
**答：** 公钥（public key）和私钥（private key）

**问：** 通过指令生成的私钥和公钥，分别怎么处理？
**答：** 私钥不用管，自动读取。公钥复制到gitee或者github里面，用于SSH认证。

**问：** id_rsa和id_rsa.pub这两个文件里面分别存放什么？
**答：** 私钥和公钥。

**问：** 生成SSH密钥对时，公钥（public key）和私钥（private key）分别放到哪里？
**答：**
- **私钥：** 保存在你的本地计算机上，必须妥善保管，不能泄露给他人
- **公钥：** 可以公开，将其上传到GitHub等远程服务器，用于验证你的身份

**问：** 当你在本地执行git push或git pull等操作时，Git会自动使用公钥还是私钥进行身份验证？
**答：** 私钥

**问：** 在git里面，配置用户名和邮箱的作用是什么？
**答：** 提交记录、代码审查、责任追踪里面会显示作者信息（用户名和邮箱）。

**问：** 克隆操作里面通过SSH链接来进行，是否需要git命令本地用户名和邮箱配置？需要什么配置？
**答：** 不需要本地用户名和邮箱配置，需要生成公钥和私钥，并将公钥配置到gitee。

**问：** Gitee完整利用SSH认证的过程？
**答：**
1. 你连接Gitee服务器
2. Gitee发送一个随机字符串
3. 你的电脑用本地私钥对这个字符串进行签名
4. Gitee用存储的公钥验证签名
5. 验证成功则允许访问

**问：** 对于SSH项目clone克隆方式，必须事先怎么样？
**答：** 在Gitee配置SSH公钥。

### 连接方式比较
**问：** HTTPS和SSH方式，这两种方式连接远程仓库，他们的效果上，区别是什么？
**答：**
- **HTTPS方式：** 如果你使用HTTPS协议连接远程仓库，推送代码时可能需要输入用户名和密码
- **SSH方式：** 如果你使用SSH密钥进行认证，则无需每次输入密码

**问：** HTTPS和SSH，这两种方式，他们分别的用途是不是不一样？HTTPS只能克隆下来，无法push，而SSH，既可以clone又可以push？
**答：** 不对，HTTPS和SSH都可以进行clone和push操作，它们的主要区别在于认证方式。HTTPS每次和平台gitee的交互都需要使用账号和密码，而SSH则不需要。

**问：** 使用HTTPS克隆命令失败且忘记账号密码怎么办？
**答：** 可切换为SSH克隆方式，但需先在网站上配置好本机的SSH公钥，系统会自动通过密钥配对完成认证。

### 项目管理
**问：** 右键菜单中的选项，Git Bash Here，作用是什么？
**答：** 在当前选定的文件夹路径下打开一个Git Bash命令行窗口，省去了在命令行中逐层进入目标目录的步骤，直接在需要的目录下开始工作。

**问：** .idea是什么文件夹？
**答：** 这是IntelliJ IDEA项目的配置文件夹。

**问：** .env文件是一个简单的文本文件，其中包含了什么变量，存放什么信息。这个文件应该被添加到哪个文件中，避免公开？
**答：** 环境变量，敏感信息（密码，密钥），应该被添加到.gitignore文件中。

**问：** .gitignore文件一般在哪里？
**答：** 项目的根目录。

**问：** 在.gitignore文件中，添加什么内容来忽略.idea文件夹？
**答：** `.idea/`

## 8. 团队协作和工作流

### Pull Request
**问：** PR（Pull Request）是一个分支吗？
**答：** 不是，而是一个请求机制。

**问：** PR是一个什么样的申请？
**答：** 请求合并代码的申请。

**问：** 将一个分支的更改合并到另一个分支（通常是主分支）的正式请求，叫做什么？
**答：** PR（Pull Request）

**问：** 为什么团队开发里面，PR是必要的？
**答：** 确保代码经过审查，来确保代码质量，团队成员可以相互学习。

**问：** Pull Request是针对谁？针对fork还是branch分支？
**答：** Pull Request可以用于fork间，也可以在同一仓库的分支间。

### 分支合并操作
**问：** Merge 'origin/group05-Scott' into 'group05'这个合并操作是本地还是远程进行的？
**答：** 本地。

**问：** Merge 'origin/group05-Scott' into 'group05'将哪里的内容合并到哪里的目标分支？
**答：** 远程分支合并到本地目标分支。

**问：** 分支合并后，为什么必须Push？
**答：** 因为合并操作是本地的，需要push到远程才能让其他人看到合并结果。

**问：** 目前已经到了目标分支group05，而右下角的显示界面，有两个group05-Scott工作分支，一个是local，另一个是remote，这个时候，选择哪个？以及为什么？
**答：** 选择远程分支，因为远程分支代表了团队可见的正式版本。

**问：** Merge 'origin/group05-Scott' into 'group05'这个选项的作用是什么？
**答：** 分支合并，将Scott这个工作分支合并到group05这个主分支。

**问：** Merge 'origin/group05-Scott' into 'group05'这个选项的出现，意味着，当前处于哪个分支？
**答：** group05分支。

**问：** 工作分支和目标分支之间的合并方向是什么？
**答：** 从工作分支合并到目标分支。

**问：** 如果需要合并功能分支，那么需要处于哪个分支？
**答：** 目标分支（主分支）。

**问：** PyCharm中的使用图形界面合并分支group05以及group05-Scott，怎么操作？
**答：** 切换到目标分支（group05），点击要合并的分支（group05-Scott），选择"Merge 'group05-Scott' into 'group05'"。

### 团队协作最佳实践
**问：** 每天开始工作前，或在开始新的任务前，先干什么git操作？
**答：** 从远程仓库拉取最新的更改。git pull。

**问：** 每日开始工作时的Git流最佳实践是什么样的？
**答：**
1. 检查状态：`git status`
2. 处理本地修改：`git add .` -> `git commit -m "..."`
3. 拉取远程更新：`git pull --rebase`
4. 推送本地修改：`git push`

**问：** 开始一天的工作前，Git操作的完整工作流是什么？
**答：**
1. 运行`git status`
2. 执行`git add .`
3. 执行`git commit -m "..."`
4. 执行`git pull --rebase`
5. 执行`git push`

**问：** 团队协作中使用Git的黄金法则是什么？
**答：** 提交后先拉再推（commit，pull --rebase，push）。

**问：** Git执行pull操作最安全、最理想的前提是什么？
**答：** 当status命令返回"nothing to commit, working tree clean"时。

**问：** 为什么在执行git pull前要先提交(commit)本地更改？
**答：** 确保本地工作已经记录在了本地仓库，这样拉取的时候才能自动合并。

**问：** 为什么在从远程仓库拉取代码前，需要先确保工作目录是干净的？
**答：** 因为若有未提交修改直接拉取，Git会尝试合并远程和本地代码，容易引发合并冲突。

**问：** 在从远程仓库拉取代码前，为确保工作目录干净，需要执行哪些Git指令？
**答：**
```bash
git add .
git commit -m "feat: complete feature A"
```

**问：** 使用git pull --rebase拉取更新后，GitHub上的版本会被改变吗？
**答：** 不会。git pull --rebase只拉取并在本地变基，只有执行git push才会更新GitHub。

### Git历史和版本管理
**问：** Git提交的唯一标识是什么？
**答：** 提交哈希值。

**问：** 查看当前分支的提交哈希值可以用哪些指令？
**答：**
- `git rev-parse --short HEAD`
- `git log -1 --oneline`
- `git branch -v`

**问：** git rev-parse --short HEAD中的rev、parse、HEAD各是什么意思？
**答：**
- **rev：** 是revision的缩写，表示"版本"
- **parse：** 表示"解析"引用字符串
- **rev-parse：** Git的内部解析工具，用于解析版本号或引用
- **HEAD：** 表示当前所指向的提交

**问：** git log -1 --oneline里面的log、-1、--oneline分别是什么意思？
**答：**
- **log：** 日志
- **-1：** 最近1条
- **--oneline：** 一行显示

**问：** 使用两台设备操作同一个仓库时，是否理想状态是两边分支的哈希一致？以及为什么？
**答：** 是的。两端的分支哈希一致，表示代码版本完全同步，是理想状态。

**问：** 两台设备上同一个分支的提交哈希值完全一样，那么意味着什么？
**答：** 那么这两个仓库的代码内容就100%同步。

### 克隆和下载
**问：** git clone里的--depth 1参数有什么作用？
**答：** 只克隆最新一次提交，减少下载量和时间。

**问：** git clone里的--depth 1参数什么时候使用？
**答：** 在只需要项目最新代码、不需要完整提交历史时使用，如快速拉取、构建或部署。

**问：** git clone里的--depth 1参数有什么好处？
**答：** 能大幅减少下载时间和磁盘占用。

**问：** 使用git clone时，不加上--depth 1为什么慢、占用更大？
**答：** 因为它拉取完整历史，包括所有提交、分支、标签和文件的历史版本。

**问：** 批量下载十几个github仓库，最佳方式是什么？
**答：** 把所有git clone指令写进一个bash脚本中，再通过bash clone_repos.sh一次执行全部下载。

### 虚拟环境和Python相关
**问：** 按照requirements.txt文件安装所有依赖的指令是什么？
**答：** `pip install -r requirements.txt`

**问：** pip install -r requirements.txt中的-r是什么意思？
**答：** -r是--requirement的缩写，表示从指定的requirements文件中读取并一次性安装所有依赖包。

**问：** 如果项目没有requirements.txt，如何生成？
**答：** 使用指令：`pip freeze > requirements.txt`

**问：** 创建名为career_env、使用Python 3.12的虚拟环境指令是什么？
**答：** `conda create -n career_env python=3.12`，其中：-n career_env指定环境名称，python=3.12指定Python版本。
